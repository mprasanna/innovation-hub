The platform decision

PWA first. React Native when push demands it.

The morning briefing → mission control → approve/clear loop is a reading and decision UI. No camera, no biometrics, no sensors. That's the whole brief for a PWA. The one concrete trigger to graduate is push notification reliability — not customer count.

Start here

Next.js PWA

One codebase. iOS and Android immediately. No App Store approval cycle blocking weekly iterations. Hosted on Catalyst or Vercel — push to deploy.

  • One codebase for mobile app + admin console + backend
  • Same auth, same data, same deployment — no API boundary to maintain
  • No App Store review: ship on Monday, execs see it on Tuesday
  • iPhone + Android immediately without a second build
  • The core loop (briefing → review → act) is pure information UI — no native APIs needed
  • Fastest path to the first 10–100 paying customers
The entire admin console we've already built lives at /admin — same app, same deploy, same login.
Graduate to this

React Native

The right choice once iOS push notification reliability becomes a business problem — not before. Everything in this product works as a well-built PWA first.

  • APNs (Apple's push service) is more reliable than iOS PWA web push
  • Biometric authentication (Face ID) for enterprise security requirements
  • Background sync and true offline for unreliable network environments
  • Native calendar/contacts integration if the product expands there
  • Enterprise MDM deployment (required by some corporate IT policies)
Trigger to graduate: the 7am briefing push notification fails to land reliably on iOS, and no amount of service-worker tuning fixes it. That's the signal.
The one thing to stress-test early: iOS PWA push has improved significantly since iOS 16.4 (Safari web push via APNs), but it still requires the user to add the app to their home screen, and it's less reliable than native push in airplane/low-signal scenarios. Test this with real executives in month 2 or 3, before the morning briefing habit depends on it.

Holds up
PWA first — core loop is information UI. No native APIs required for V1.
One codebase, /admin route — exactly matches the admin console we built.
4 AI functions — consistent with the lean build plan.
Events table as source of truth — Mission Control, Briefing, and Timeline all derive from it.
No agents, no LangChain, no AutoGen — direct API calls are simpler, debuggable, and sufficient for V1.
One correction
Use PostgreSQL, not Catalyst Data Store for production data. Catalyst Data Store has row and query limits that will bite you. Use Neon or Supabase (hosted Postgres) from day one — same simplicity, no ceiling.
Everything else in the stack is current and appropriate for the scale target.
Watch closely
iOS PWA push notifications. Test delivery reliability with real devices in month 2. The morning briefing habit depends on the notification landing. If it doesn't, that's your React Native trigger.
"No agents initially" doesn't mean no agents ever. Once Month 3 risk detection runs, the natural next step is an autonomous risk-response loop. Design the 4 functions with that in mind — keep them composable.
The real challenge

The hardest part isn't the technology. It's getting executives to open the morning briefing every day.

The 2–3/10 stack exists specifically so you can iterate on the habit loop fast — change the briefing format, adjust the attention score weights, test notification timing — without fighting your own infrastructure. Build the simplest thing that can demonstrate Morning Briefing → Mission Control → All Clear. That's the product.
