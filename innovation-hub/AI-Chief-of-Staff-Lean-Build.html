The recommendation

Six pieces. That's the entire stack.

Everything in one repository. No service mesh, no broker, no orchestration layer to operate. The simplicity is the feature — it's what lets one person own the whole system.

Frontend
Next.js
The UI and the app shell, server-rendered where it helps.
Backend
Next.js API routes
Same codebase. No separate server to deploy or reason about.
Database
PostgreSQL
One relational store. Add pgvector later if you want recall.
AI
OpenAI / Claude
Behind four functions. Swap providers without touching the app.
Auth
Auth.js (NextAuth)
Login and the Google OAuth scopes you need are the same flow.
Deployment
Vercel
Push to deploy. Add a background worker only when jobs grow.
Deliberately not in V1
NestJS Redis Temporal Kafka Microservices Kubernetes

For the first ten customers, a single Next.js app and one Postgres database handle everything. The complexity comes later — and only if growth earns it.

The data model

One append-only event stream. Everything reads from it.

The single most important table is events. Approvals, meetings, risks and check-ins all emit events — and Mission Control, the Briefing and the Timeline are just different views over the same stream.

The AI layer

Four functions. That's enough for Version 1.

The model handles language work — summarising, recommending, explaining. The business logic stays in plain code. Everything routes through these four entry points.

generateBriefing()

Reads the event stream and composes the ranked morning brief.

evaluateApproval()

Checks an approval against policy and recommends approve / review.

summarizeMeeting()

Turns a transcript into minutes, decisions and action items.

analyzeRisk()

Compares stated plans against assigned work to flag deviation.

Attention score

Don't over-engineer it. Start at 100 and adjust.

The same idea as the scale track's scoring engine, stripped to its core: a handful of deductions and additions over a baseline. Store the result. Tune the weights once you've watched it on real days.

Scoring · illustrative weightsbase = 100
Critical risk−20
Customer escalation−15
Approval pending−10
Risk mitigated+10
Approval resolved+5
All of it, in code
let score = 100
score += criticalRisk      ? -20 : 0
score += approvalPending   ? -10 : 0
score += customerEscalation? -15 : 0
score += approvalResolved  ? +5  : 0
score += riskMitigated     ? +10 : 0
store(score)
Done.
What to build first

Four milestones — each one a loop you can demo.

Same sequence as the scale track, compressed: prove one closing loop at a time. Read “week” as “milestone” — and mock the integrations early so the pace is real.

Week 1
Approval intelligence
approval→AI rec→approve→audit log

The first closing loop. Fetch, recommend, decide, record.

Week 2
Meeting minutes
transcript→summary→actions

Transcript in, structured minutes and action items out.

Week 3
Team risk
assigned+daily update→mismatch→risk

Detect the gap between stated plan and assigned work.

Week 4
Morning briefing
approvals+risks+meetings→briefing

Compose the three loops into one ranked daily brief.

The key realization
Database+ Integrations+ LLM+ UI

The prototype makes it look like a massive system. Technically, the first version is just those four things. The complexity comes later — and only if you earn it. Spend the next sprint on the event model, not more screens.
